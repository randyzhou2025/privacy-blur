# PrivacyBlur MVP 方案文档

## 1. 项目定位

项目名称暂定：PrivacyBlur

一句话定位：

> 一个可以在浏览器本地运行的晒图隐私遮挡器，图片不上传服务器，只在用户自己的设备里完成识别、确认、打码和导出。

面向场景：

- 小红书晒聊天截图前，遮挡昵称、头像、手机号、地址、订单号。
- 晒快递、订单、票据、报名信息前，遮挡姓名、电话、地址、编号。
- 晒旅行、生活照前，手动遮挡车牌、证件、孩子学校、二维码等敏感区域。

第一版目标不是做“大而全图片编辑器”，只解决一个痛点：

> 用户想发图，但怕隐私泄露；工具帮他快速发现可疑隐私区域，并把选中的区域真正打码到导出图片里。

## 2. MVP 范围

第一版采用“方案 A + 方案 B”：

- 方案 A：纯手动本地打码
- 方案 B：本地 OCR + 规则识别

第一版必须包含：

- 图片本地选择与预览。
- 手动框选隐私区域。
- 本地 OCR 识别截图中的文字。
- 对手机号、邮箱、身份证号、订单号、金额、姓名字段、地址关键词等做规则匹配。
- 把自动识别结果作为“候选遮挡区域”展示给用户确认。
- 用户可新增、移动、删除遮挡框。
- 导出一张已经写入遮挡效果的新图片。
- 页面明确显示“图片不上传，只在本机处理”。

第一版暂不做：

- 云端 OCR。
- 云端 AI 识图。
- 用户账号。
- 历史记录。
- 图片云存储。
- 批量处理。
- 完整图片编辑功能。
- AI 自动修图、美化、滤镜。

## 3. 隐私原则

产品最大的信任点不是“识别很准”，而是：

> 用户不需要相信服务器，因为图片根本不离开用户设备。

必须遵守以下原则：

1. 图片文件不上传服务器。
2. OCR 在浏览器本地运行。
3. 不调用第三方云端识图、OCR、大模型接口。
4. 不保存原图、OCR 文本、遮挡区域、导出图。
5. 不使用第三方统计 SDK。
6. 不读取图片 EXIF 以外的无关信息。
7. 导出图片时重新编码，尽量去掉原图 EXIF 元数据。
8. 用户关闭页面后，内存中的图片和 OCR 结果自然清空。

推荐页面信任文案：

> 图片仅在你的浏览器本地处理，不上传、不存储、不训练模型。自动识别只是提醒，导出前请手动确认是否还有遗漏。

## 4. 设计与开发第一准则

后续设计、开发、审查和测试都必须把用户体验优先作为第一准则。

具体执行要求：

- 页面第一屏优先服务用户当前任务，不把营销说明、次要设置或说明文字放在主要操作前面。
- 每个控件的位置、大小和状态都要从用户的连续操作路径出发，例如“选择图片 -> 看到图片 -> 查找/手动打码 -> 导出”。
- 开发时不能只以功能可用为验收标准，还要检查是否容易误触、是否节省操作、是否符合手机端单手使用和小屏阅读。
- 审查 UI 改动时优先看真实页面截图或浏览器视口效果，而不是只看代码结构。
- 测试时必须覆盖移动端关键视口，重点检查首屏信息层级、按钮位置、文字是否拥挤、编辑手势是否顺手。

## 5. 核心用户流程

主流程：

```text
打开网页
  -> 选择图片
  -> 浏览器本地读取图片
  -> 显示图片预览
  -> 用户可以手动框选隐私区域
  -> 用户点击“自动查找隐私文字”
  -> 本地 OCR Worker 识别文字
  -> 规则匹配手机号/邮箱/地址/订单号/金额/姓名等
  -> 生成候选遮挡框
  -> 用户确认/删除/调整候选框
  -> 点击导出
  -> Canvas 把遮挡效果写入新图片
  -> 生成新图片下载
```

关键体验：

- 自动识别结果必须是“候选”，不要替用户直接定稿。
- 导出前给用户一个清晰检查状态，例如“已遮挡 8 处，仍建议检查头像、车牌、二维码等非文字隐私”。
- 手动框选必须足够顺滑，因为它是隐私兜底能力。

## 6. 技术架构

建议第一版做成纯前端静态应用：

```text
静态页面
  -> File API 读取本地图片
  -> Canvas 渲染和导出
  -> Web Worker 跑 OCR
  -> 本地规则引擎匹配隐私文本
  -> 用户确认遮挡框
```

推荐技术栈：

- Vite + TypeScript
- Canvas 2D
- Tesseract.js 或同类浏览器 OCR
- Web Worker
- CSS 原生实现界面
- 后续可加 PWA 离线缓存

项目结构建议：

```text

    main.ts

    ocr-worker.ts

    models/
    tessdata/
```

## 6. 方案 A：纯手动本地打码

### 6.1 功能目标

用户上传图片后，可以直接用鼠标或手指在图片上画框，然后选择遮挡样式并导出。

### 6.2 遮挡样式

第一版建议只保留 2 种：

- 纯色遮挡：最安全，默认推荐。
- 马赛克遮挡：更自然，但安全性弱于纯色遮挡。

默认使用纯色遮挡，例如黑色或深灰色。

不要只做页面浮层遮挡，必须在导出时把遮挡效果写入 Canvas 像素。

### 6.3 核心实现

```text
input[type=file]
  -> createImageBitmap(file)
  -> canvas.drawImage(bitmap)
  -> 用户拖拽生成 boxes
  -> 预览层显示 boxes
  -> 导出时重新绘制原图和 boxes
  -> canvas.toBlob()
```

遮挡框数据结构：

```ts
type MaskBox = {
  id: string;
  x: number;
  y: number;
  width: number;
  height: number;
  source: "manual" | "ocr";
  label?: string;
  confidence?: number;
};
```

注意：

- 框坐标必须以原图坐标保存，不要只保存屏幕显示坐标。
- 页面缩放时，需要做 display 坐标和 image 坐标换算。
- 导出前重新绘制，避免预览缩放影响清晰度。

## 7. 方案 B：本地 OCR + 规则识别

### 7.1 功能目标

对截图中的文字进行本地识别，找出疑似隐私信息并生成候选遮挡框。

第一版重点识别：

- 手机号
- 邮箱
- 身份证号
- 银行卡号疑似
- 订单号/快递单号疑似
- 金额疑似
- 地址关键词
- 姓名字段附近文本

### 7.2 OCR 流程

```text
图片
  -> 降采样或压缩到适合 OCR 的尺寸
  -> 传给本地 OCR Worker
  -> 获取文字块和坐标
  -> 合并相邻文字块
  -> 规则匹配
  -> 映射回原图坐标
  -> 生成候选遮挡框
```

### 7.3 规则示例

```ts
const privacyRules = [
  {
    type: "手机号",
    pattern: /(?<!\d)1[3-9]\d{9}(?!\d)/g,
    risk: "high",
  },
  {
    type: "邮箱",
    pattern: /[A-Z0-9._%+-]+@[A-Z0-9.-]+\.[A-Z]{2,}/gi,
    risk: "high",
  },
  {
    type: "身份证号",
    pattern: /\b\d{17}[\dXx]\b/g,
    risk: "high",
  },
  {
    type: "银行卡号疑似",
    pattern: /\b\d{16,19}\b/g,
    risk: "medium",
  },
  {
    type: "订单号疑似",
    pattern: /\b[A-Z0-9-]{10,}\b/gi,
    risk: "medium",
  },
  {
    type: "地址疑似",
    pattern: /(省|市|区|县|镇|乡|路|街|号|小区|栋|幢|单元|室)/g,
    risk: "medium",
  },
];
```

地址、订单号、金额和姓名不要宣称精准识别，只显示为“疑似”。

### 7.4 候选框生成

如果 OCR 返回的是 word 级坐标：

- 命中单个词：遮挡该 word 的 bounding box。
- 命中一整段文本：遮挡相关 line 的 bounding box。
- 地址类关键词：遮挡所在整行，并提示用户检查上下文。

遮挡框需要向外扩展一点：

```text
x -= 8
y -= 6
width += 16
height += 12
```

这样可以避免文字边缘泄露。

### 7.5 OCR 的限制

必须在页面中提醒用户：

- 自动识别可能漏掉头像、二维码、车牌、学校名、背景文字等非规则隐私。
- 中文 OCR 可能识别错误。
- 自动识别结果仅供提醒，导出前请手动检查。

## 8. 导出安全

导出必须重新生成图片，而不是下载原图叠加层。

导出流程：

```text
创建离屏 Canvas
  -> 按原图尺寸绘制原图
  -> 逐个绘制遮挡框
  -> canvas.toBlob("image/png" 或 "image/jpeg")
  -> 创建 Object URL
  -> 触发下载
  -> revokeObjectURL
```

导出建议：

- PNG：适合截图，默认格式。
- JPEG：适合照片，可选质量 0.92。
- 导出的图片不保留原始 EXIF。
- 文件名使用 `privacyblur-masked-时间戳.png`。

## 9. 网络与隐私防护

第一版上线时建议做到：

- 所有前端文件自托管。
- OCR wasm 和语言包自托管。
- 不使用 CDN。
- 不接 Google Analytics、百度统计、友盟、神策等 SDK。
- 不上传图片。
- 不上传 OCR 文本。
- 不上传错误日志中的图片内容。

推荐安全响应头：

```nginx
add_header Content-Security-Policy "default-src 'self'; img-src 'self' blob: data:; worker-src 'self' blob:; script-src 'self'; style-src 'self' 'unsafe-inline'; connect-src 'self'; object-src 'none'; base-uri 'self'; frame-ancestors 'none';" always;
add_header Referrer-Policy "no-referrer" always;
add_header X-Content-Type-Options "nosniff" always;
```

如果后续需要基础访问统计，只统计页面访问，不记录图片、文件名、OCR 文本、遮挡框坐标。

## 10. 页面结构建议

第一屏直接是工具，不做营销落地页。

页面区域：

```text
顶部：
  PrivacyBlur
  图片不上传，只在本地处理

左侧/上方：
  上传图片
  自动查找隐私文字
  手动遮挡
  导出图片

中间：
  图片编辑画布

右侧/下方：
  已发现的隐私候选
  已遮挡数量
  清空全部
  隐私说明
```

关键按钮：

- 选择图片
- 自动查找隐私文字
- 添加遮挡
- 删除选中
- 导出图片

关键提示：

> 自动识别仅供提醒，请在导出前检查头像、二维码、车牌、学校名等非文字隐私。

## 11. 第一版验收标准

功能验收：

- 可以上传 PNG/JPG/WebP。
- 图片不会被发送到网络请求中。
- 可以手动画框遮挡。
- 可以运行本地 OCR。
- 可以识别手机号、邮箱、身份证号、订单号疑似、金额疑似、姓名疑似、地址疑似。
- 可以把 OCR 命中的区域显示为候选框。
- 用户可以删除或调整候选框。
- 导出图片中已经包含遮挡效果。
- 刷新页面后不保留原图和处理记录。

隐私验收：

- 打开浏览器 Network 面板，处理图片时没有图片上传请求。
- 断网后，已加载完成的页面仍可进行手动打码。
- 没有第三方统计请求。
- 没有把 OCR 文本写入 localStorage、IndexedDB 或服务器日志。

体验验收：

- 移动端可用。
- 大图预览不卡死。
- OCR 运行时有进度提示。
- 用户能清楚知道哪些区域会被导出遮挡。
